## System Design: JWT Authentication with Cookie Storage

### Task
Design a JWT-based authentication system with tokens stored in HTTP-only cookies for a microservices architecture.

### Context
- Existing codebase: Go backend (API Gateway + Auth Service), Next.js frontend
- Current state: Login exists but no JWT tokens, no cookie storage
- Architecture: DDD in auth service, modular in API Gateway, gRPC communication

### Design Completed
**File**: `../project-context/jwt-cookie-authentication-design.md`

**Key Design Decisions**:
1. **Token Strategy**: 
   - Access token: 15 min, stateless (no DB lookup)
   - Refresh token: 7 days, stateful (Redis blacklist)

2. **Security Approach**:
   - HTTP-only cookies (prevent XSS)
   - SameSite=Strict (prevent CSRF)
   - Secure flag in production (HTTPS only)
   - Separate Redis DB (index 1) for token blacklist

3. **Architecture**:
   - JWT service in auth service domain layer
   - Token repository interface + Redis implementation
   - JWT middleware in API Gateway
   - Cookie-setting at API Gateway layer

4. **New Components**:
   - JWT Service (domain layer)
   - Token Repository (domain + infrastructure)
   - Refresh, Logout, Validate use cases
   - JWT middleware (API Gateway)
   - Cookie-setting controllers (API Gateway)

5. **Trade-offs Made**:
   - Cookies > Authorization header (security priority)
   - Stateless access + stateful refresh (performance balance)
   - 15 min access token (security > convenience)
   - HS256 > RS256 (internal-only validation)
   - Redis > PostgreSQL for blacklist (performance)
